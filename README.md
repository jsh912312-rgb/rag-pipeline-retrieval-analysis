1. **프로젝트 개요**

대학 입시 모집요강 PDF를 파싱하여 사용자의 자연어 질문에 정확하게 답변하는 RAG(Retrieval-Augmented Generation) 시스템을 구축한 프로젝트입니다. 모집인원, 수능최저, 제출서류, 지원자격, 평가방법 등 5개 정보 유형에 대해 질문-답변 정확도를 평가합니다.

**평가 지표 요약**

| 지표 | 설명 |
| --- | --- |
| MRR (Mean Reciprocal Rank) | 정답 청크가 검색 결과 몇 번째에 나타나는지의 평균 역순위. 1에 가까울수록 정답이 최상위에 노출됨 |
| Hits@K | 상위 K개 결과 안에 정답이 포함된 비율 |
| nDCG@K | 순위 감쇠를 반영한 검색 품질 지표. 정답이 앞에 위치할수록 높은 점수 |
| AnswerAccuracy | LLM이 생성한 답변이 실제 정답과 일치하는 비율 |
| NoAnswerRate | LLM이 “모르겠다”고 응답한 비율 (낮을수록 좋음) |

1. **데이터 버전별 문서 구조**

**2-1. v2- 페이지 단위 청킹 + 메타데이터 보강**

**총 문서 수: 2,749개**

V2는 PDF 페이지를 의미 단위로 분할한 뒤, 다양한 메타데이터 필드를 부착하는 방식으로 설계되었습니다.

**스키마**

```json
{
  "doc_id":          "연세대학교_수시_010",
  "university":      "연세대학교",
  "admission_type":  "수시",
  "track":           "특기자",
  "section":         "전형개요",
  "page":            7,
  "has_nonsul":      0,
  "has_suneung":     0,
  "text_length":     536,
  "flag_short_chunk": false,
  "keywords":        "학년도, 단계, 모집단위, 구분, 교과",
  "content":         "원문 텍스트 (표 포함 복합 구조)",
  "is_header_chunk": false,
  "has_dept_data":   true,
  "dept_list":       "컴퓨터과학과, 첨단컴퓨팅학부, ...",
  "enriched_content": "[연세대학교] [수시] [특기자전형] [전형개요]\n원문\n\n[전형별 모집인원 요약]\n연세대학교 수시 컴퓨터과학과 모집인원 총 66명 ..."
}
```

**주요 특징**

- `section` 필드로 전형개요 / 제출서류 / 지원자격 / 수능최저 / 평가방법 등을 구분
- `enriched_content`: 태그 접두사(`[대학] [전형] [유형]`)를 원문 앞에 붙여 임베딩 품질 향상 시도
- `has_dept_data` + `dept_list`: 모집인원 관련 청크를 식별하기 위한 플래그
- `keywords`: TF-IDF 등으로 추출한 핵심 단어를 메타데이터로 추가
- 하나의 청크가 여러 학과 정보를 혼합하여 포함하는 경우가 많음 (표 기반 데이터 특성)

**section 분포 (2,749개)**

```
전형개요:  1,416개  (51.5%)
제출서류:    429개
지원자격:    428개
유의사항:    248개
평가방법:     93개
수능최저:     89개
전형방법:     28개
기타:         10개
전형일정:      8개
```

**2-2. V3 — 원자 단위 정제 청킹**

**총 문서 수: 1,121개**

V3는 V2의 복합 구조 청크 문제를 해소하기 위해, 하나의 청크가 **하나의 정보 사실**만을 담도록 설계했습니다.

**스키마**

```json
{
  "doc_id":         "강원대학교_수시_0001",
  "university":     "강원대학교",
  "admission_type": "수시",
  "track":          "교과",
  "major":          "멀티디자인학과",
  "info_type":      "모집인원",
  "content":        "강원대학교 수시 교과전형 멀티디자인학과 모집인원은 48명이다."
}
```

**주요 특징**

- `info_type` 필드로 질문 유형과 문서 유형을 직접 매핑 (모집인원 / 지원자격 / 제출서류 / 수능최저 / 평가방법)
- `major` 필드로 학과 단위 검색 필터링 가능
- content가 단일 사실 문장 형태로 정규화되어 임베딩 정밀도 향상
- 불필요한 메타데이터 필드 제거로 스키마 단순화 (16개 → 7개 필드)
- 전형개요·유의사항 등 직접 답변과 무관한 섹션 제거

**info_type 분포 (1,121개)**

```
모집인원:   980개  (87.4%)
지원자격:    41개
제출서류:    28개
유의사항:    24개
수능최저:    17개
평가방법:    16개
기타:        10개
전형일정:     5개
```

**2-3. V2 vs V3 비교 요약**

| 항목 | V2 | V3 |
| --- | --- | --- |
| 총 문서 수 | 2,749개 | 1,121개 |
| 청킹 단위 | PDF 페이지 / 의미 단위 | 단일 사실 (원자 단위) |
| 스키마 필드 수 | 16개 | 7개 |
| 핵심 분류 필드 | `section` (섹션명 문자열) | `info_type` (정규화된 카테고리) |
| 학과 정보 | `dept_list` (콤마 나열) | `major` (단일 값) |
| content 형태 | 원문 텍스트 (표 포함, 복수 정보 혼합) | 단일 사실 문장 |
| 임베딩 대상 | `enriched_content` (태그 + 원문) | `content` (사실 문장 자체) |
| 메타데이터 필터 | section 기반 | info_type + major 기반 |
- **핵심 차이**
    - V2는 원문 구조를 최대한 보존하되 메타데이터로 보강하는 방식
    - V3는 원문을 재구조화하여 검색 목적에 최적화된 단일 사실 문장으로 변환하는 방식

**3. RAG 파이프라인 구성**

```
[원본 PDF]
    │
    ▼
[문서 파싱 및 청킹]  ── V2: 페이지 단위 + overlap
    │                ── V3: 원자 단위 정제
    ▼
[임베딩 생성]        ── 배치 임베딩 (FAISS 저장)
    │
    ▼
[Vector DB 저장]     ── 메타데이터 포함 색인
    │
    ▼
[Retriever]          ── 필터(메타데이터) + Semantic 검색 혼합
    │
    ▼
[Prompt 구성]        ── 검색된 컨텍스트 + 질문 조합
    │
    ▼
[LLM 답변 생성]
    │
    ▼
[평가]               ── 검색 품질 (MRR / Hits@K / nDCG)
                     ── 답변 품질 (AnswerAccuracy / NoAnswerRate)
```

- **필터 전략:** 질문에서 대학명, 수시/정시, 정보 유형을 추출하여 메타데이터 필터로 적용 후 semantic 검색 수행

**5. 오류 분석**

**5-1. 모집인원 — 학과 특정 실패 (41개 케이스, Hits@10=0)**

- 패턴 예시
    
    ```
    Q   : 한림대학교 사회복지학부 수시 모집인원 수는?
    정답 : 한림대학교_수시_1338, 한림대학교_수시_1319
    검색 : 한림대학교_수시_1247, 한림대학교_수시_1040, 한림대학교_수시_1026
    ```
    
- 원인 분석
    
    V2에서는 하나의 청크에 여러 학과 모집인원이 표 형태로 섞여 있다. “사회복지학부”를 담은 청크의 임베딩이 “사회복지학부”라는 단어 하나보다 페이지 전반의 문맥에 의해 결정되므로, 유사 구조를 가진 다른 청크와의 벡터 거리 차이가 미미하다. 같은 대학·전형이지만 다른 페이지 번호를 가진 청크들이 상위에 노출된다.
    
- V3에서 개선된 이유
    - `강원대학교 수시 교과전형 멀티디자인학과 모집인원은 48명이다.`처럼 학과명과 모집인원이 하나의 문장에 명시되므로, `major` 필터와 semantic 검색 모두 정확하게 작동한다.

**5-2. 지원자격 — 전형 내 청크 혼재 (7개 케이스, Hits@10=0)**

- 패턴 예시
    
    ```
    Q   : 한양대학교 수시 학종전형 지원자격 조건은?
    정답 : 한양대학교_수시_1838, 한양대학교_수시_1835
    검색 : 한양대학교_수시_1770, 한양대학교_수시_1789, 한양대학교_수시_1765
    ```
    
- 원인 분석
    
    V2의 `section` 필드가 “전형개요”와 “지원자격”을 구분하지만, 실제로 모집요강에서는 지원자격 내용이 전형개요 섹션 안에 혼재되어 있는 경우가 많다. 결과적으로 전형 공통 정보를 담은 청크(1770번대)가 학종전형 특화 지원자격 청크(1835번대)보다 높은 유사도를 가지게 된다.
    

**5-3. 제출서류 — 전형 식별 모호성 (3개 케이스)**

- 패턴 예시
    
    ```
    Q   : 연세대학교 수시 특기자전형 제출서류 목록은?
    정답 : 연세대학교_수시_178, 연세대학교_수시_160
    검색 : 연세대학교_수시_100, 연세대학교_수시_035, 연세대학교_수시_021
    ```
    
- 원인 분석
    
    연세대학교 수시 문서에서 공통 안내(035, 021번)와 전형별 제출서류(160, 178번)가 별개 청크로 분리되어 있으나, 공통 안내 청크도 “제출서류”라는 표현을 포함하고 있어 semantic 검색에서 혼동이 발생한다.  `track` 필터가 “특기자”로 제대로 작동하지 않거나, 청크 자체에 “특기자”라는 단어가 희소하게 등장하는 경우이다.
    

**5-4.  수능최저 — 전형 단위 매핑 오류**

- 패턴 예시
    
    ```
    Q   : 강원대학교 수시 학종전형 수능 최저학력기준은?
    정답 : 강원대학교_수시_760, 강원대학교_수시_758
    검색 : 강원대학교_수시_591, 강원대학교_수시_601, 강원대학교_수시_513
    ```
    
- 원인 분석
    
    수능최저 기준은 전형 공통으로 기술되는 경우와 모집단위별로 기술되는 경우가 혼재한다. 검색된 청크(591, 601번)는 같은 수시 전형이지만 다른 섹션(전형개요 또는 모집인원 표)에 해당하며, 수능최저 내용을 포함하지 않는다.

1. **개선 방안**

**6-1. 표 청크 재구성 — table_schema 도입**

모집인원 파싱 실패의 근본 원인인 표 유형 혼재 문제를 해결합니다.

- 표 유형 분류 후 분리 구조화
    
    ```
    표 유형 분류
    ├── 학과표     → table_schema: { 학과, 인원, 전형 }  ← 모집인원 청크
    ├── 감점표     → table_schema: { 항목, 감점점수 }    ← 별도 색인 또는 제외
    └── 비율표     → table_schema: { 전형요소, 반영비율 } ← 평가방법 청크
    ```
    

학과표만 모집인원 청크로 색인하고, 감점표·비율표는 각자의 `info_type`으로 분리 구조화하거나 검색 대상에서 제외한다. 이를 통해 모집인원 청크의 임베딩이 학과명·인원 수치에 집중되도록 한다.

**6-2. 섹션 레이블 강화 — 2단계 계층 구조**

단일 `section` 레이블을 `section_lv1 / section_lv2` 2단계로 세분화합니다.

```json
{
  "section_lv1": "지원자격",
  "section_lv2": "졸업요건"     // 또는 "교과이수조건", "기타조건"
}
```

| section_lv1 | section_lv2 | 설명 |
| --- | --- | --- |
| 지원자격 | 졸업요건 | 고졸(예정)자 등 기본 학력 조건 |
| 지원자격 | 교과이수조건 | 특성화고·직업계 등 이수 과목 조건 |
| 지원자격 | 기타조건 | 농어촌·재직자·장애인 특별전형 조건 |
| 제출서류 | 공통서류 | 전형 무관 공통 제출 서류 |
| 제출서류 | 전형별서류 | 특정 전형에만 해당하는 제출 서류 |

`section_lv2` 필터를 추가하면 “학종전형 지원자격” 쿼리에서 졸업요건 청크와 교과이수조건 청크를 분리하여 더 정밀한 검색이 가능합니다.

**6-3. Dual-Index 하이브리드 구조 설계**

현재 단일 인덱스에 모든 정보 유형이 혼재하는 구조를 개선합니다.

설계 방향

```
[질문 입력]
    │
    ▼
[질문 유형 분류기]  → info_type 예측 (모집인원 / 지원자격 / 제출서류 / ...)
    │
    ├─ 모집인원 → Index A (학과표 특화 색인, BM25 + Dense 하이브리드)
    │              필터: university + admission_type + major
    │
    └─ 지원자격/제출서류/수능최저/평가방법
           → Index B (규정문서 특화 색인, Dense + section_lv2 필터)
                필터: university + admission_type + track + section_lv2
```

- 기대 효과
    - 모집인원처럼 고유명사(학과명) 매칭이 중요한 유형은 BM25가 강점을 발휘하는 Index A로 라우팅
    - 지원자격처럼 긴 조건문의 의미 파악이 필요한 유형은 Dense 검색 중심의 Index B로 라우팅
    
     → 이를 통해 전체 필터 의존도를 분산하고, 필터 추출 실패 시에도 유형별 특화 인덱스가 보조 역할        을 합니다.
    
- 필터 추출 Fallback 전략 (단기 적용 가능)

```
1차: 필터(대학 + 전형 + info_type + section_lv2) + Semantic
2차 fallback: 필터(대학 + info_type) + Semantic
3차 fallback: 필터(대학) + Semantic
```

**7. 한계점**

- 평가셋 규모 불균형
    - v3 평가가 25개 케이스로밖에 수행되지 않아 통계적 신뢰성이 낮다
- 표 데이터 파싱 한계
    - 모집인원 정보는 대부분 표의 형태로 구성되어 있다.
    - pdf에서 표를 파싱하면 셀 순서가 뒤섞이거나 행/열 구분이 소실되어 나온다.
- 필터 의존도 문제
    - 대학명이나 전형명등을 생략하거나 비표준 표현을 사용하는 경우 필터 추출이 실패할 수 있다.
- LLM 품질의 병목
    - QA 답변 정확도가 검색 품질보다 낮다. 이는 검색에 성공하더라도 LLM이 올바른 수치나 조건을 추출하지 못하는 경우가 존재한다는 것이다.
    - 관련 도메인 특화 fine-tuning 혹은 모델로의 교체가 필요하다.
